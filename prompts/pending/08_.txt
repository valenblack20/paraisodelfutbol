==================================================
1. AUDIT AND FIX AUTH FOUNDATIONS FIRST
==================================================

Before adding CRUD, audit:

- middleware;
- CSRF lifecycle;
- login, logout and session endpoints;
- AuthService;
- admin shell;
- product and category schema;
- repository contracts;
- cart stock behavior;
- current CSP;
- all uses of `any`.

Fix these issues before implementing writes:

A. CSRF persistence

The CSRF token must not depend only on sessionStorage after login.

Implement one safe approach:

- server-render a raw CSRF token into authenticated admin pages after validating the session; or
- create an authenticated endpoint that rotates and returns a new CSRF token.

Requirements:

- store only the token hash in MySQL;
- allow the admin application to recover a valid token after page reload;
- rotate safely;
- never expose session tokens;
- require CSRF for POST, PUT, PATCH and DELETE;
- use timing-safe comparison.

B. Middleware scope

Only query and validate admin sessions for:

- `/admin`
- `/admin/*`
- `/api/admin/*`

A stale admin cookie must not cause database work or failures on public pages.

Handle authentication infrastructure errors safely:

- browser admin routes: controlled Spanish unavailable page or 503;
- admin API routes: JSON 503;
- public routes: remain unaffected.

C. Type safety

Replace every `catch (error: any)` with `unknown` and explicit narrowing.

D. CSP

Remove `unsafe-eval` if the current Astro output works without it.
Reduce `unsafe-inline` where practical.
Document any directive that must remain.

E. Login request protection

Validate Origin or Referer for login POST requests against the configured site origin.
Reject invalid cross-origin login requests.

==================================================
2. DATA CONSISTENCY DECISION
==================================================

`product_variants.stock` becomes the authoritative inventory source.

Administrators must edit stock only through variants.

`products.stock` must not be independently editable.

Choose and implement one documented compatibility strategy:

Preferred:

- keep `products.stock` temporarily;
- update it inside the same transaction as variant writes;
- set it to the sum of active variant stocks.

Never allow the total product stock and variant stock to diverge after an admin operation.

==================================================
5. TRANSACTIONS
==================================================

All multi-table product writes must use a MySQL transaction.

For create or update:

1. begin transaction;
2. validate category exists;
3. insert or update product;
4. insert/update variants;
5. insert/update images;
6. calculate total active variant stock;
7. update products.stock;
8. commit.

On failure:

- rollback;
- return safe error;
- never leave partial variants or images;
- never expose SQL details.

Use a dedicated connection from the pool for the transaction.
Always release it in `finally`.

==================================================
6. CONCURRENCY CONTROL
==================================================

Prevent accidental lost updates.

Add a migration:

008_add_product_version.sql

Add to products:

- version INT UNSIGNED NOT NULL DEFAULT 1

On update:

- client sends expectedVersion;
- update uses:
  `WHERE id = ? AND version = ?`;
- increment version atomically;
- if affectedRows = 0, return HTTP 409.

Admin message in Spanish:

"El producto fue modificado por otra sesión. Recargá los datos antes de volver a guardar."

Extend the same strategy to categories only if needed.
At minimum, products must have optimistic concurrency.

==================================================
7. ARCHIVING INSTEAD OF HARD DELETE
==================================================

Do not permanently delete products through the first admin version.

Add migration:

009_add_product_archiving.sql

Fields:

- archived BOOLEAN NOT NULL DEFAULT FALSE
- archived_at TIMESTAMP NULL

Rules:

- archived products are excluded from public queries;
- archived products remain visible in the admin with a filter;
- archiving sets published = false;
- restoring does not automatically publish;
- variants and images remain stored;
- public detail returns 404 for archived products.

Do not expose hard DELETE for products.

Categories referenced by products cannot be deleted.
Use active/inactive state instead.

==================================================
14. IMAGE AND VARIANT UPDATE STRATEGY
==================================================

For this initial admin version, replacing the complete collection is acceptable inside a transaction.

Variants:

- validate all first;
- upsert existing rows by ID where appropriate;
- delete removed rows only inside the transaction;
- preserve IDs for unchanged variants when possible;
- recalculate products.stock.

Images:

- validate all first;
- ensure exactly one primary;
- preserve IDs where possible;
- delete removed image rows inside the transaction;
- synchronize products.primary_image_path with the selected primary image.

Never leave primary_image_path pointing to a removed image.
